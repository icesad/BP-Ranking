# 阶段 1 开发清单 · 分类评分卡估值 + 改进建议（2026-06-15）

> 配合 `2026-06-15-vibecoder-home-roadmap.md` 阶段 1 蓝图。按顺序做，每步可勾选、带验收点。
> 在本地现有栈(better-sqlite3)上做；**沙箱跑不了 better-sqlite3，所有验收在本机 `npm run dev` 后自查**。
> 改了表结构要**重启 dev server** 让迁移生效。

## 关键设计约定（先读）

- **rubric 是一次项目级 `deepseek-chat` 调用**（像现有 `valuationLevers`/`deepseekClassify`），不进 12 位投资人 reasoner 循环——省钱省时。
- 12 家估值仍是"市场区间"主体；rubric 当结构化解释层 + 驱动建议 + 决定口径(regime)。
- 守红线：缺数据维度给保守低分并标"未知"；**最终仍输出区间+置信度**；倍率有界，不无脑连乘；C(成本)只当弱下限。

## Step 1 · DB 迁移与字段（lib/db.js）

- [ ] `migrations` 数组追加（幂等，老库无损）：
  - `ALTER TABLE bps ADD COLUMN archetype TEXT DEFAULT ''`（website/game/tool/saas/ai_agent/ecommerce/community/other）
  - `ALTER TABLE valuations ADD COLUMN algo_version TEXT DEFAULT 'v1'`
  - `ALTER TABLE valuation_history ADD COLUMN algo_version TEXT DEFAULT 'v1'`
  - `ALTER TABLE outcomes ADD COLUMN verified INTEGER DEFAULT 0`（0=未核实/创始人声称，1=有来源已核实）— 自报数据不当主干，仅可核实者抬 floor
- [ ] `CREATE TABLE IF NOT EXISTS suggestions`：
  `id PK, bp_id, archetype, dim, rubric_dim, title, detail, effort, impact_low REAL, impact_high REAL, evidence_url, algo_version, created_at`
  - `dim` ∈ feature/differentiation/avoid/visual/ux/growth/coldstart/tech/moat
  - `rubric_dim` ∈ completeness/validation/commercial/maintainability/market/moat
  - 加 `CREATE INDEX idx_sugg_bp ON suggestions(bp_id)`
- 验收：重启 dev server 无报错；`sqlite3` 看 `PRAGMA table_info(bps)` 有 archetype、`suggestions` 表存在。

## Step 2 · 原型分类 archetype（lib/engine.js + lib/deepseek.js）

- [ ] `lib/engine.js`：加 `ARCHETYPE_LABELS`(中英) + `classifyArchetypeFallback(bp)`（按关键词/demo_type 兜底，如 demo_type=html 倾向 website/game/tool）；`normalizeClassification` 接受并清洗 `archetype`。
- [ ] `lib/deepseek.js` `deepseekClassify`：prompt 的 JSON 增字段 `"archetype":"从[website,game,tool,saas,ai_agent,ecommerce,community,other]选一个"`。
- [ ] 写库处（upload/upload-demo/rescore 调分类的地方）把 archetype 落到 bps；分类失败用 fallback。
- 验收：新上传一个 demo，详情页/数据库能看到合理 archetype；无 key 时 fallback 也给值。

## Step 3 · 六维评分卡 rubric（lib/deepseek.js 新函数）

- [ ] 新增 `async function scoreRubric(bp, archetype, evidence, comparables, floor)`：一次 `deepseek-chat`，按该 archetype 的"重点信号"给六维打分(0–100)，每维必须带证据点，缺证据给保守分并 `known:false`。
  - 输出 JSON：`{dims:{completeness:{score,evidence,known},validation:{...},commercial:{...},maintainability:{...},market:{...},moat:{...}}, en:{...}}`
  - prompt 要写明：缺数据从保守、团队不计分、不要用平台级公司类比抬分（沿用现有 valuateDemo 的诚实原则）。
- [ ] 导出 `scoreRubric`；无 key 返回 null（上层降级：用 simEvaluation 的维度近似或全部保守）。
- [ ] `lib/engine.js` 加 `ARCHETYPE_WEIGHTS`（六维权重表，每行合计 100，见路线图阶段 1 表）+ `compositeScore(dims, archetype)` → 0–100 加权分。
- 验收：对一个已估值项目手动调用，返回六维分 + 证据；游戏类 validation/moat 权重高、saas 类 commercial 高。

## Step 4 · 单一主干估值 + 可观测信号优先（lib/valuation.js）

> **不做"成本 vs 收入"二选一的 regime**。理由：用户多半不愿填收入、填了难核实、不会回来更新 → 依赖自报收入的路径默认空/过期，且违背诚实原则。永远算同一条主干，自报收入只是可选、需证据、随时间衰减的输入。

- [ ] 采集**第一方可观测信号** `collectSignals(db, bp)`（最可信，自动、不靠用户更新）：
  - 站内 demo：试玩/play 次数、停留时长、复访(`visits`)、关注、分享（埋点见 Step 9b）。
  - URL/GitHub：stars/最近提交/响应速度（可比库管线已有，随 tick 刷新）。
- [ ] 调 `scoreRubric(bp, archetype, evidence, comps, floor, signals)`：validation/commercial 维度**优先采用 signals**，自报数据仅在 `verified=1` 或带来源时给小权重并按 `occurred_at` 时间衰减。
- [ ] 主干：`compositeScore` → **有界倍率**映射成对该 archetype 可比区间的相对位置（设上下限，禁止无界连乘）→ floor 兜底。可核实的融资/收入只**抬 floor + 加 commercial 证据分**，不另起公式。
- [ ] 综合 summary 增字段：`archetype`、`rubric`(六维分+证据)、`compScore`、`signals`(第一方信号快照)。保留现有 low/high/confidence/disp/floor/levers/compList。
- 验收：无收入项目也能正常出区间、不被赛道热度抬飞；只填"未核实"自报收入**不**明显改变估值；加一条**带来源(verified=1)**的 raised outcome 后重估，floor 抬升、low≥floor；半年前的旧自报数据权重明显低于近期。

## Step 8 · 改进建议引擎（lib/deepseek.js + API + 写库）

- [ ] 新增 `async function improvementPlan(bp, rubric, archetype, comparables, valuation)`：依据**评分卡弱项** + 可比库"强竞品"信号，产出 4–6 条建议，每条：`{dim, rubric_dim, title, detail, effort, impact_low, impact_high, evidence_url, en}`。
  - impact 用**有界**幅度（如 +5%~+30%，或绝对额区间），不夸大；avoid/differentiation 由"竞品太强"信号触发。
- [ ] 新增 `app/api/suggest/[id]/route.js`：触发 improvementPlan 并写 `suggestions` 表（覆盖旧版，带 algo_version）。可在 valuateBp 成功后顺带触发，或按钮手动触发（省钱）。
- 验收：对一个项目生成建议，弱维度(低分那维)确实出现对应建议；impact 区间合理不离谱。

## Step 9c · 第一方信号的公开展示策略（详情页 + lib）

> 决策：**公开展示，但做成"防刷 + 不羞辱冷启动"**，不裸露原始计数。原则：展示偏质量/相对/趋势，刷量无收益(自然消除动机)，小样本不出丑。

- [ ] **展示什么**：优先**质量/相对指标**——复访率、留存、停留时长、近 7 日趋势(↑/↓)、相对同原型分位(如"试玩活跃度前 20%")；弱化易刷的**原始计数**(试玩总数等可放次要位或只给量级)。
- [ ] **小样本保护**：低于阈值(如 < N 次试玩/访客)时显示"🌱 数据积累中"，不显示难看的小数字；达到样本量再亮指标。
- [ ] **防刷(与估值红线一致)**：
  - 按 visitor 去重(已有 `visits` 每访客每天一条)、忽略作者本人/自访、限频，尽量过滤 bot。
  - **喂给估值的是去重过滤后的信号**——刷量基本抬不动估值，从根上削动机。
  - 异常激增打标、可疑不计入估值。
- [ ] **作者私有视图 vs 公开视图**：作者(阶段2有账户后)在个人页看完整明细；公开页只露上面这套经过筛选的子集。
- 验收：新项目显示"数据积累中"不出丑；同一访客反复刷新计数不涨；公开页看不到原始可刷大数、只看相对/质量指标。

## 备注 / 风险

- rubric + 建议都靠 LLM，**成本控制**：rubric 每次 1 个 chat 调用、建议手动触发为主；沿用 valHash 跳过。
- 评分卡权重是**起始值**，必须由 Step 6 回测对齐真实结果后再调——科学性来自命中率，不是公式好看。
- 沙箱测不了 better-sqlite3；写完由你在本机验收，或迁到 Cloudflare(阶段 0)后在 D1 上测。
